Vehicle Tracker — System Design

TJTS5901 · Continuous Software Engineering · Course Project

Scenario 2: User picks a bus route from a dropdown → sees live bus positions on a map → updates automatically
Frontend — Browser What the user sees and interacts with No business logic here
templates/index.html
  • Page shell — loads Leaflet.js and app.js
  • Route dropdown — populated via /api/routes on load
  • Map container div — Leaflet attaches here
  • Status bar — shows freshness and bus count
  • Error banner — shown when feed is stale
static/app.js
  • Initialises Leaflet map centred on Jyväskylä
  • Opens WebSocket to /ws/{route_id} on route select
  • On message: update or create bus markers
  • Rotating arrow icon — direction from bearing field
  • Auto-reconnect if WebSocket drops
  • Popup on click: bus label, speed, next stop
static/style.css
  • Layout: header bar + full-height map
  • Status badge colours: green/yellow/red
  • Minimal — map does the heavy lifting
HTTP — GET /api/routes  |  WebSocket /ws/{route_id}
Application Layer — FastAPI Serves endpoints and pushes data to clients. No logic, no HTTP calls to Waltti. Only calls Processing layer
app/main.py
  • Creates FastAPI app instance
  • Mounts static files and templates
  • Registers routers
  • Startup event: connects to Redis
  • CORS middleware for local dev
app/routes.py
  • GET / → serve index.html
  • GET /api/routes → list of route IDs from live feed
  • WebSocket /ws/{route_id} → accept connection, subscribe to Redis Pub/Sub channel, push new positions to this client whenever worker publishes
  • On client disconnect: unsubscribe cleanly
app/dependencies.py
  • Redis client singleton (shared across requests)
  • Injected into route handlers via FastAPI Depends()
function calls — no HTTP
Processing Layer Pure Python — takes raw data in, returns clean results out. No HTTP, no Redis, no UI. Only called by Application. Only calls Ingestion.
processing/vehicle_service.py
  • filter_by_route(vehicles, route_id) → list
  • get_all_route_ids(vehicles) → sorted list of unique IDs
  • compute_freshness(fetched_at) → LIVE / DELAYED / STALE + age label
processing/models.py
  • Vehicle dataclass: id, route_id, lat, lon, bearing, speed_kmh, label, timestamp
  • FreshnessStatus dataclass: level, age_seconds, label
  • FreshnessLevel enum: LIVE / DELAYED / STALE
function calls — no HTTP
Ingestion Layer Fetches from Waltti. Decodes protobuf. Writes to Redis. Knows nothing about routes or the UI. Only touches Waltti API and Redis
ingestion/worker.py
  • Runs as a separate process (not inside FastAPI)
  • Infinite loop: sleep 30s → fetch → decode → write Redis → publish
  • On fetch failure: log error, skip write, keep old Redis data alive
  • Exponential backoff on repeated failures
ingestion/feed_fetcher.py
  • fetch_vehicle_positions() → raw bytes from Waltti GTFS-RT
  • Sets x-api-key header from config
  • 10s timeout, raises on non-200 response
  • Called only by worker.py
ingestion/gtfs_parser.py
  • parse_feed(raw_bytes) → list of Vehicle objects
  • Uses gtfs-realtime-bindings to decode protobuf
  • Skips entities with no position field
  • Null-checks every optional field (bearing, speed)
ingestion/cache_writer.py
  • write_to_redis(vehicles, redis_client)
  • Stores JSON per route_id with 60s TTL
  • Publishes "vehicles:updated" Pub/Sub event
  • FastAPI subscribes to this event → pushes to WebSocket clients
Redis Cache + Pub/Sub
Cache — vehicles:{route_id}
  • Key: vehicles:4 → JSON array of vehicle positions
  • TTL: 60s (survives one missed worker cycle)
  • FastAPI reads this key when client connects
Pub/Sub — vehicles:updated
  • Worker publishes after every successful write
  • FastAPI subscribes — one message → push to all connected WebSocket clients
  • No polling needed on the server side
External APIs Not our infrastructure
Waltti GTFS-RT API
  • Vehicle positions feed — binary protobuf
  • Called once per 30s by worker — never by FastAPI
  • Auth: x-api-key header from .env
  • URL: configured via WALTTI_BASE_URL env var
OpenStreetMap Tiles
  • Map background — fetched directly by browser
  • Free, no API key required
  • Leaflet handles tile loading automatically
Configuration
config.py
  • Loads all settings from environment via python-dotenv
  • WALTTI_BASE_URL, WALTTI_API_KEY
  • REDIS_URL, CACHE_TTL, STALE_THRESHOLD
  • Single import used across all layers
.env  /  .env.example
  • .env — real secrets, never committed to git
  • .env.example — template with placeholder values, committed
  • WALTTI_API_KEY=your_key_here
  • REDIS_URL=redis://localhost:6379

How Data Flows — Step by Step
1
User opens the app
Browser loads index.html. Leaflet map appears centred on Jyväskylä. JavaScript calls /api/routes which reads from Redis (or triggers a Waltti fetch if cache is empty). Dropdown is populated with active route IDs.
Browser→ GET /api/routes→ routes.py→ vehicle_service.py→ Redis read→ JSON → dropdown
2
User selects Route 4
Browser opens a WebSocket to /ws/4. FastAPI reads vehicles:4 from Redis and immediately sends current positions. Client draws bus markers on the map. Connection stays open.
Browser WS → /ws/4→ routes.py→ Redis read vehicles:4→ send JSON→ app.js → draw markers
3
Worker runs every 30 seconds (separate process)
worker.py wakes up, calls feed_fetcher.py which fetches from Waltti. gtfs_parser.py decodes the protobuf binary. cache_writer.py stores updated positions in Redis and publishes "vehicles:updated". FastAPI receives the Pub/Sub event and pushes new positions to all connected WebSocket clients. Browsers update markers — no polling from client side.
worker.py→ feed_fetcher.py → Waltti→ gtfs_parser.py → Vehicle list→ cache_writer.py → Redis write + publish→ routes.py → WS push→ Browser updates
4
Waltti API is unavailable
feed_fetcher.py raises an exception. worker.py catches it, logs the error, skips the Redis write. Redis TTL keeps old data alive for up to 60s. FastAPI serves stale positions with freshness level = STALE. Browser shows the red error banner. No crash, data still visible.
feed_fetcher.py → timeout→ worker.py catches exception→ Redis TTL still valid → stale data served→ freshness = STALE → red banner
