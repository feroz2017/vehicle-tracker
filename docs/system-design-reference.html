Ingestion Worker — Kubernetes Deployment (replicas: 1 + leader election)
Polls Waltti GTFS-RT every 30s · Decodes protobuf · Produces to Kafka · Never calls Waltti more than once per 30s regardless of user count. Kubernetes Deployment ensures auto-restart if crashed. Kubernetes Lease API for leader election if scaled to 2 replicas (only leader polls). Exponential backoff on Waltti failures (1s → 2s → 4s → max 60s).
Leader Election · Auto-restart · Backoff
Kubernetes makes everything self-healing. Pod crashes → restarted in seconds. HPA adds FastAPI pods when load spikes — no manual intervention. Leader election means exactly one worker polls Waltti even if the deployment has 2 replicas. Graceful shutdown prevents mid-flight WebSocket connections from being dropped hard.

Read Replica 1 & 2: history queries, never touches primary.

PgBouncer · Primary + 2 Replicas · Fallback
Three separate data stores, each doing what it's best at. Redis: sub-millisecond current state. Kafka: durable event log, decouples producer from consumers, enables replay. PostgreSQL: permanent history and complex queries. If any one fails, the others keep the system running.

Resilience Patterns
Circuit breaker — if Redis latency >50ms, open circuit, serve from DB
Retry + backoff — worker retries Waltti: 1s, 2s, 4s, 8s…max 60s
Dead letter queue — Kafka DLQ for messages that fail processing 3 times
Graceful shutdown — FastAPI drains WebSockets on SIGTERM before pod dies
Liveness + readiness probes — k8s kills and replaces unhealthy pods
PodDisruptionBudget — always keep ≥2 FastAPI pods running during node maintenance
Design for failure. Assume every component will fail. Build so it doesn't matter.

5 Critical Data Flows
1
User loads page — static files served from Cloudflare edge. Origin server never touched. Zero load.
Browser→ Cloudflare CDN edge→ cached → index.html + map.js served instantly
2
User selects Route 4 — WebSocket opened, FastAPI reads Redis (microseconds), pushes positions. No Waltti call, no DB query.
Browser WSS→ Kong GW→ FastAPI pod→ read → Redis→ JSON push → map markers drawn
3
Worker fetches fresh data (every 30s) — produces to Kafka. Kafka consumer writes Redis + Postgres. Redis publishes Pub/Sub. All FastAPI pods push to their WebSocket clients simultaneously.
Worker→ protobuf → Waltti API→ produce → Kafka→ consume → Redis write + publish→ all FastAPI pods→ all clients update
4
Redis fails — circuit breaker opens. FastAPI falls back to PostgreSQL read replica. Slower (5–10ms vs <1ms) but data is served. Alertmanager pages on-call engineer. Sentinel promotes a replica to master within 30s.
FastAPI→ Redis timeout → circuit opens→ PgBouncer → Postgres replica→ data served, banner shown+ PagerDuty alert fired
5
FastAPI pod crashes — Kubernetes detects via liveness probe failure (<10s). Pod replaced. Nginx Ingress stops routing to it immediately. New pod registered in <30s. WebSocket clients reconnect automatically (client-side retry logic).
FastAPI pod→ crash → k8s liveness probe fails→ Ingress removes pod→ new pod starts→ clients auto-reconnect
