Are WebSocket connections stored in Redis?
No — and they cannot be. A WebSocket is a live TCP connection object in memory. It cannot be serialised or stored in Redis.

What Redis provides instead is Pub/Sub coordination. Each FastAPI instance keeps its own in-memory dictionary: { route_id → [ws_client_1, ws_client_2, ...] }. When the worker publishes to the vehicles:updated channel, every FastAPI instance receives that event simultaneously and each pushes the update to its own locally connected WebSocket clients.

The load balancer uses sticky sessions (consistent hashing on client IP or a session cookie) so each client stays connected to the same FastAPI instance for the lifetime of their WebSocket connection. If that instance dies, the client reconnects — the browser-side code retries automatically — and the load balancer routes them to a healthy instance.

1
User types "Jyväskylä station" → "University of Jyväskylä" and hits Search
FastAPI checks geocoding cache for both addresses. On miss: calls Digitransit Geocoding API → gets lat/lon → stores in Redis (24h TTL). Then checks route plan cache. On miss: calls Digitransit GraphQL with coordinates → gets list of itineraries → stores in Redis (5 min TTL) → returns to browser.
Browser POST /api/plan→ FastAPI→ cache miss → Geocoding API → lat/lon→ Digitransit GraphQL → routes→ Redis write plan:{hash}→ JSON routes → Browser
2
Same search again (within 5 minutes)
FastAPI checks Redis → cache hit → returns immediately. No Digitransit call, no geocoding call.
Browser POST /api/plan→ FastAPI → Redis hit→ JSON routes instantly
3
User selects a route — WebSocket opens
Browser opens WebSocket to /ws/vehicles/{route_id}. FastAPI immediately reads current positions from Redis and pushes — client sees buses on the map right away. FastAPI then subscribes to the Redis Pub/Sub channel and waits for worker updates.
Browser WS /ws/vehicles/4→ FastAPI accept→ Redis read vehicles:4→ push JSON → map draws+ subscribe vehicles:updated
4
Worker fires every 30 seconds
Worker fetches Waltti, decodes protobuf, writes all vehicle positions to Redis, publishes one event. All FastAPI instances receive the event via Pub/Sub. Each reads Redis and pushes to its own connected WebSocket clients. Every browser updates simultaneously — no polling from client side.
Worker → Waltti API→ decode protobuf→ Redis write vehicles:{id}→ publish vehicles:updated→ FastAPI 1 + FastAPI 2 receive→ all WS clients updated
5
FastAPI Instance 1 crashes
Load balancer detects failure via health check. Clients on Instance 1 lose their WebSocket. Browser-side reconnect logic kicks in after 3 seconds — reconnects to Instance 2. FastAPI 2 reads current state from Redis and resumes pushing. No data is lost because state is in Redis, not in the crashed instance.
Instance 1 crash→ LB removes it→ Browser reconnects after 3s→ Instance 2 → reads Redis → resumes push
